# IEC 62304:2006+AMD1:2015 – Medizingeräte-Software – Lebenszyklusprozesse

## Normsteckbrief

| Merkmal | Beschreibung |
|---------|-------------|
| **Nummer** | IEC 62304:2006+AMD1:2015 |
| **Titel** | Medical device software – Software life cycle processes |
| **Herausgeber** | International Electrotechnical Commission (IEC) |
| **Erstveröffentlichung** | 2006, Amendment 1: 2015 |
| **Anwendungsbereich** | Definiert Lebenszyklusprozesse für die Entwicklung von Medizingeräte-Software und Software innerhalb von Medizinprodukten |
| **Relevanz für MeDoc** | **Best Practice** – verpflichtend nur bei Einstufung als Medizinprodukt gemäß EU MDR 2017/745 |

## Inhalt und Struktur der Norm

IEC 62304 definiert einen vollständigen Software-Lebenszyklus für medizinische Software. Die Norm gliedert sich in folgende Prozessbereiche:

### 1. Allgemeine Anforderungen (Kapitel 4)
- **Qualitätsmanagementsystem**: Der Hersteller muss ein QMS betreiben (typischerweise nach ISO 13485).
- **Risikomanagement**: Risikomanagement nach ISO 14971 muss in den Software-Lebenszyklus integriert sein.
- **Software-Sicherheitsklassifizierung**: Jedes Software-System muss einer Sicherheitsklasse zugeordnet werden.

### 2. Software-Sicherheitsklassen

IEC 62304 definiert drei Sicherheitsklassen, die den erforderlichen Dokumentationsumfang bestimmen:

| Klasse | Beschreibung | Beispiel |
|--------|-------------|---------|
| **Klasse A** | Kein Beitrag zu einer Gefährdungssituation möglich, ODER Gefährdung wird durch Schutzmaßnahmen außerhalb der Software kontrolliert | Verwaltungssoftware ohne Einfluss auf Behandlung |
| **Klasse B** | Beitrag zu einer Gefährdungssituation möglich, die zu **nicht-schwerwiegender** Verletzung führen kann | Software zur Terminplanung mit Medikamentenwarnungen |
| **Klasse C** | Beitrag zu einer Gefährdungssituation möglich, die zu **schwerwiegender Verletzung oder Tod** führen kann | Dosierungsberechnungssoftware, Strahlentherapie-Steuerung |

**Einordnung MeDoc:** Sollte MeDoc als Medizinprodukt eingestuft werden, wäre **Klasse A oder B** wahrscheinlich, da das System keine direkte Therapiesteuerung durchführt, aber klinische Befunddaten verwaltet, deren fehlerhafte Darstellung zu einer Fehlentscheidung führen könnte.

### 3. Software-Entwicklungsprozess (Kapitel 5)

Die Norm verlangt folgende Aktivitäten, abhängig von der Sicherheitsklasse:

| Aktivität | Klasse A | Klasse B | Klasse C |
|-----------|:--------:|:--------:|:--------:|
| Softwareentwicklungsplanung | ✅ | ✅ | ✅ |
| Anforderungsanalyse | ✅ | ✅ | ✅ |
| Architekturentwurf | – | ✅ | ✅ |
| Detailentwurf | – | – | ✅ |
| Unit-Implementierung | ✅ | ✅ | ✅ |
| Unit-Verifikation | – | ✅ | ✅ |
| Integrations- und Integrationstest | – | ✅ | ✅ |
| Systemtest | ✅ | ✅ | ✅ |
| Softwarefreigabe | ✅ | ✅ | ✅ |

### 4. Software-Wartungsprozess (Kapitel 6)
- Wartungsplan erstellen
- Problem- und Änderungsanalyse durchführen
- Änderungen implementieren und verifizieren

### 5. Software-Risikomanagementprozess (Kapitel 7)
- Analyse von Software-Beiträgen zu Gefährdungssituationen
- Risikokontrollmaßnahmen definieren und verifizieren
- Risikomanagement bei Softwareänderungen

### 6. Software-Konfigurationsmanagement (Kapitel 8)
- Konfigurationsidentifikation (alle Artefakte versioniert)
- Änderungskontrolle (Change Control)
- Konfigurationsstatus-Dokumentation

### 7. Software-Problemlösungsprozess (Kapitel 9)
- Problemberichte erstellen und verfolgen
- Problemursachen untersuchen
- Relevante Parteien informieren
- Korrekturen über Änderungskontrollprozess implementieren
- Trendanalyse durchführen

### SOUP (Software of Unknown Pedigree / Provenance)

IEC 62304 definiert besondere Anforderungen an **SOUP** – Software unbekannter Herkunft, d.h. Drittanbieter-Komponenten und Open-Source-Bibliotheken:
- Jede SOUP-Komponente muss identifiziert und dokumentiert werden
- Risikobewertung der SOUP-Nutzung erforderlich
- Bekannte Anomalien und deren Auswirkungen müssen bewertet werden

**Relevanz für MeDoc:** MeDoc verwendet zahlreiche SOUP-Komponenten: React, Tauri, SQLite, sqlx, bcrypt, und weitere npm/crate-Pakete. Eine SOUP-Liste mit Risikobewertung ist Best Practice.

| ISO-62304-01 | Ein Software-Entwicklungsplan muss erstellt und gepflegt werden | Kap. 5.1 | MUST |
| ISO-62304-02 | Software-Anforderungen müssen formal dokumentiert und nachverfolgbar sein | Kap. 5.2 | MUST |
| ISO-62304-03 | Die Software-Architektur muss dokumentiert werden und die Umsetzung der Anforderungen nachweisen | Kap. 5.3 | MUST |
| ISO-62304-04 | Alle Software-Einheiten müssen implementiert und verifiziert werden | Kap. 5.5, 5.6 | MUST |
| ISO-62304-05 | Systemtests müssen die Erfüllung der Anforderungen nachweisen | Kap. 5.7 | MUST |
| ISO-62304-06 | Ein formaler Softwarefreigabeprozess muss definiert sein | Kap. 5.8 | MUST |
