# ISO/IEC 25010:2011 – Software-Qualitätsmodell (SQuaRE)

## Normsteckbrief

| Merkmal | Beschreibung |
|---------|-------------|
| **Nummer** | ISO/IEC 25010:2011 |
| **Titel** | Systems and software engineering – Systems and software Quality Requirements and Evaluation (SQuaRE) – System and software quality models |
| **Herausgeber** | ISO/IEC (Joint Technical Committee JTC 1/SC 7) |
| **Erstveröffentlichung** | 2011 (ersetzt ISO/IEC 9126) |
| **Anwendungsbereich** | Definiert ein Qualitätsmodell mit 8 Hauptmerkmalen und 31 Untermerkmalen zur Bewertung der Softwarequalität |
| **Relevanz für MeDoc** | **Relevant** – bietet strukturierte Qualitätskriterien für die Anforderungsdefinition und Qualitätssicherung |

## Das Qualitätsmodell nach ISO 25010

ISO/IEC 25010 definiert **8 Qualitätshauptmerkmale** mit insgesamt **31 Untermerkmalen**:

### 1. Funktionale Eignung (Functional Suitability)

Grad, in dem die Software Funktionen bereitstellt, die die festgelegten und vorausgesetzten Bedürfnisse erfüllen.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Funktionale Vollständigkeit** | Alle geforderten Funktionen sind abgedeckt | 45+ funktionale Anforderungen (FA-TERM bis FA-STAT), Traceability-Matrix |
| **Funktionale Korrektheit** | Ergebnisse sind korrekt und genau | Validierungspflicht für Patientenakten (Arzt-Freigabe), Konfliktprüfung bei Terminen |
| **Funktionale Angemessenheit** | Funktionen unterstützen die Aufgaben angemessen | Praxis-Workflows abgebildet (Termin → Untersuchung → Behandlung → Abrechnung) |

### 2. Leistungseffizienz (Performance Efficiency)

Leistung relativ zu den eingesetzten Ressourcen.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Zeitverhalten** | Antwort- und Verarbeitungszeiten | NFA-PERF-01 (<2s pro Aktion), Fuzzy-Suche <500ms |
| **Ressourcenverbrauch** | Speicher- und CPU-Verbrauch | Tauri (leichtgewichtig), SQLite (embedded, geringer Footprint) |
| **Kapazität** | Maximale Datenmengen und gleichzeitige Nutzer | SQLite WAL für 3-5 gleichzeitige Nutzer, Pagination für große Datenmengen |

### 3. Kompatibilität (Compatibility)

Fähigkeit, Informationen auszutauschen und/oder Funktionen mit anderen Systemen zu teilen.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Koexistenz** | Betrieb neben anderer Software ohne Beeinträchtigung | Desktop-App als eigenständiger Prozess (Tauri), keine Portbelegung nötig |
| **Interoperabilität** | Datenaustausch mit anderen Systemen | NFA-INT-01 (Röntgen-Import), NFA-INT-02 (Drucker), PDF-Export (FA-AKTE-06) |

### 4. Benutzbarkeit (Usability)

Grad, in dem die Software von Benutzern effektiv, effizient und zufriedenstellend verwendet werden kann.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Erkennbarkeit der Eignung** | Benutzer erkennt, ob Software für seine Aufgabe geeignet ist | Sidebar-Navigation mit klarer Kategorisierung, Dashboard-Übersicht |
| **Erlernbarkeit** | Aufwand zum Erlernen der Benutzung | NFA-USE-05 (Einarbeitungszeit ≤2 Monate), Tooltips, intuitive Workflows |
| **Bedienbarkeit** | Aufwand der Bedienung | NFA-USE-02 (max. 2 Klicks zur Hauptfunktion), Drag-and-Drop |
| **Schutz vor Benutzerfehlern** | Schutz vor fehlerhaften Eingaben | NFA-USE-03 (Bestätigungsdialoge), Validierung aller Pflichtfelder |
| **Ästhetik der Benutzeroberfläche** | Ansprechende, übersichtliche Darstellung | NFA-USE-01 (Nielsen-Heuristik ≥80%), Clean UI Design |
| **Barrierefreiheit** | Nutzbarkeit für Menschen mit verschiedenen Fähigkeiten | Farbkodierung + Textlabels im Zahnschema, ausreichende Kontrastwerte |

### 5. Zuverlässigkeit (Reliability)

Grad, in dem die Software ihre Funktionen unter festgelegten Bedingungen über einen festgelegten Zeitraum erfüllt.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Reife** | Häufigkeit von Fehlern im Normalbetrieb | Rust-Backend (Speichersicherheit), strenge Typisierung (TypeScript) |
| **Verfügbarkeit** | Bereitschaft zur Nutzung bei Bedarf | SA-04 (Offline-Fähigkeit), Desktop-App (kein Server-Abhängigkeit) |
| **Fehlertoleranz** | Weiterarbeiten trotz Software- oder Hardwarefehlern | SQLite WAL (ACID), automatische Wiederherstellung nach Absturz |
| **Wiederherstellbarkeit** | Wiederherstellung von Daten nach Ausfall | NFA-SEC-05 (Tägliches verschlüsseltes Backup, Wiederherstellungsprozess) |

### 6. Sicherheit (Security)

Schutz von Informationen und Daten.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Vertraulichkeit** | Daten nur für Berechtigte zugänglich | RBAC, verschlüsselte DB (SQLCipher/AES-256), Passwort-Login |
| **Integrität** | Schutz vor unbefugter Änderung | Audit-Log, Versionierung (FA-AKTE-03), RBAC für Schreiboperationen |
| **Nichtabstreitbarkeit** | Aktionen können einem Benutzer zugeordnet werden | Audit-Log mit User-ID + Zeitstempel (NFA-SEC-04) |
| **Zurechenbarkeit** | Handlungen können eindeutig zugeordnet werden | Individuelle Benutzerkonten, kein Account-Sharing |
| **Authentizität** | Identität eines Benutzers kann nachgewiesen werden | Passwort-basierte Authentifizierung (bcrypt-Hashing) |

### 7. Wartbarkeit (Maintainability)

Aufwand für Modifikationen an der Software.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Modularität** | Änderungen in einem Modul haben minimale Auswirkung auf andere | Clean Architecture (Backend), MVC-Pattern (Frontend), 10 Tauri-Befehlsmodule |
| **Wiederverwendbarkeit** | Module können in anderen Kontexten wiederverwendet werden | Generische Repository-Schicht, gemeinsame UI-Komponenten |
| **Analysierbarkeit** | Aufwand zur Fehlerdiagnose | Strukturierte Rust-Fehlerbehandlung, Logging, TypeScript-Typsicherheit |
| **Modifizierbarkeit** | Aufwand für Änderungen | Schichtenarchitektur, Trennung von Domain/Application/Infrastructure |
| **Testbarkeit** | Aufwand zum Testen von Änderungen | Unit-testbare Domänenlogik, isolierte Repositories |

### 8. Übertragbarkeit (Portability)

Fähigkeit der Software, in andere Umgebungen übertragen zu werden.

| Untermerkmal | Beschreibung | MeDoc-Umsetzung |
|-------------|-------------|-----------------|
| **Anpassbarkeit** | Anpassung an verschiedene Umgebungen | Tauri (Cross-Platform: Windows, macOS, Linux) |
| **Installierbarkeit** | Aufwand der Installation | Tauri-Installer (MSI/DMG/AppImage), keine externe DB-Installation nötig |
| **Austauschbarkeit** | Ersetzbarkeit durch andere Software | Standard-Datenformate (CSV/JSON-Export geplant), PDF-Export |

|----|-------------|-----------------|-----------|
| ISO-25010-01 | Alle 45+ funktionalen Anforderungen müssen durch die Traceability-Matrix auf Stakeholder-Bedürfnisse rückführbar sein (Funktionale Vollständigkeit) | 1 – Funktionale Eignung | MUST |
| ISO-25010-02 | Kritische Dateneingaben müssen validiert werden; Patientenakten erfordern Arzt-Freigabe (Funktionale Korrektheit) | 1 – Funktionale Eignung | MUST |
| ISO-25010-03 | Alle Benutzeraktionen müssen in unter 2 Sekunden eine sichtbare Systemreaktion erzeugen (Zeitverhalten) | 2 – Leistungseffizienz | SHOULD |
| ISO-25010-04 | PDF- und CSV-Export müssen für relevante Daten verfügbar sein (Interoperabilität) | 3 – Kompatibilität | MUST |
| ISO-25010-05 | Die Benutzeroberfläche muss eine Bewertung ≥80% bei heuristischer Evaluation nach Nielsen erreichen (Benutzbarkeit) | 4 – Benutzbarkeit | MUST |
| ISO-25010-06 | Farbkodierungen im Zahnschema müssen zusätzlich Textlabels besitzen (Barrierefreiheit) | 4 – Benutzbarkeit | SHOULD |
| ISO-25010-07 | Das System muss offline vollständig funktionsfähig sein (Verfügbarkeit) | 5 – Zuverlässigkeit | MUST |
| ISO-25010-08 | Bei Systemabsturz dürfen keine Daten verloren gehen; SQLite WAL muss aktiv sein (Fehlertoleranz) | 5 – Zuverlässigkeit | MUST |
| ISO-25010-09 | Die Architektur muss modular aufgebaut sein: Clean Architecture (Backend) + MVC (Frontend) (Wartbarkeit) | 7 – Wartbarkeit | MUST |
| ISO-25010-10 | Die Anwendung muss als plattformübergreifender Installer bereitgestellt werden (Windows, macOS, Linux) (Installierbarkeit) | 8 – Übertragbarkeit | MUST |
